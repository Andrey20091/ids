## 1. Что это за проект

**IDS ML Project** — это учебно-демонстрационная система **обнаружения сетевых угроз** с помощью машинного обучения. Она не «лезет в ваш интернет сама»: вы подаёте на вход **записи о сетевых потоках** (файлы CSV в стиле датасета CICIDS2017), либо **собираете трафик** через локальный прокси или из PCAP. Программа:

- приводит данные к единому виду (**подготовка признаков**);
- **обучает** несколько моделей (в т.ч. Random Forest, Isolation Forest; опционально — нейросети, если установлен PyTorch);
- **прогоняет детекцию** и выдаёт **алерты** (подозрительная активность);
- умеет **дообучаться по новым данным** (online), вести **инциденты**, сравнивать модели в **песочнице** и показывать всё в **веб-дашборде**.

Это **не замена коммерческому IPS** и не гарантия «ловли всего в реальном времени на гигабитах»: это полный **сквозной пайплайн** для лабораторных работ и стендов.

**Почему в репозитории уже есть модели (`artifacts/`):** чтобы после клона можно было сразу прогнать **`detect`** (после появления `flows.csv`), не ожидая часов обучения на полном CICIDS. Обучение всё равно доступно: вы можете переобучить модели на своих данных и политике из `settings.yaml`.

**Граница ответственности:** качество детекции зависит от соответствия **признаков в CSV** тому, на чем обучены модели, и от **порогов** в конфиге. Проект даёт инструменты и проверки (`validate`, тесты), но не гарантирует «нулевой FP/FN» на произвольном трафике без настройки.

## 2. Как это работает одним абзацем

**Сырой CSV** (или NDJSON с прокси, или PCAP) → скрипт **prepare** строит таблицу признаков **`flows*.csv`** → **train / baseline-train** сохраняют модели в **`artifacts/`** → **detect** читает потоки и пишет **`storage/alerts_latest.json`** → дашборд и инциденты читают эти файлы. Параллельно **online** может подстраивать модели по накапливающемуся буферу **`flows_online_buffer.csv`**, не обязательно перезаписывая основной снимок **`flows.csv`**.

**Зачем два файла потоков (`flows.csv` и `flows_online_buffer`):** snapshot удобен для воспроизводимых прогонов и отчётов; буфер — для потока событий (прокси, инкремент), чтобы не ломать snapshot при каждой новой порции данных. В `settings.yaml` пути задаются в `paths.*`; команды `detect` / `online` / `realtime` по умолчанию опираются на эти пути.

## 3. Структура папок — что где лежит

| Папка / файл | Назначение |
|----------------|------------|
| **`main.py`** | Единая точка входа: все подкоманды (`detect`, `prepare`, `proxy`, …). Запускайте из корня проекта. |
| **`pyproject.toml`** | Метаданные пакета; в т.ч. **`requires-python`** (сейчас **от 3.11 до 3.13 включительно**, не 3.14) и зависимости для `pip install -e .`. |
| **`requirements.txt`** | Базовые зависимости (pandas, sklearn, PyYAML, …). |
| **`requirements-ml.txt`** | Опционально: PyTorch и всё для «глубоких» моделей. |
| **`.env.example`** | Пример переменных окружения (если используются в вашей среде). |
| **`run.cmd`**, **`run_utf8.cmd`**, **`run.ps1`** | Удобные обёртки запуска под Windows (в т.ч. UTF-8 для консоли). |
| **`run_*_example.cmd` / `.ps1`** | Готовые сценарии: корпоративный e2e, полный ТЗ-пятница и т.д. |

### `config/`

Настройки **без правки кода**:

- **`settings.yaml`** — пути к файлам данных (`flows.csv`, буфер онлайн, SIEM, пороги угроз, политика обучения, интервалы online и др.).
- **`feature_columns.yaml`** — какие колонки и производные признаки участвуют в ML.
- Другие YAML — вспомогательные схемы (например каноника числовых полей CICIDS).

### `data/`

- **`raw/`** — «как пришло»: синтетические демо-файлы, выгрузки CICIDS (часть нужно скачать отдельно), NDJSON прокси, PCAP-конверсии. Внутри есть маленькие **`README.md`** по подпапкам.
- **`processed/`** — результат **prepare**: основной **`flows.csv`**, большие **`flows_baseline.csv`**, **`flows_online_buffer.csv`** для потокового контура и т.п.

### `artifacts/`

**Обученные модели** и реестр:

- файлы вида **`rf_model.joblib`**, **`if_model*.joblib`**, нейросетевые веса — в зависимости от того, что вы обучили;
- **`model_registry.json`** — какая версия моделей «активна», какие наборы есть для sandbox / deploy.

Не коммитьте сюда гигантские бинарники, если политика git это запрещает; локально это рабочие файлы.

### `storage/`

**Рабочее состояние приложения** (алерты, истории, чекпоинты):

- **`alerts_latest.json`** — последние алерты для дашборда;
- **`incidents.jsonl`**, **`incident_actions.jsonl`** — инциденты и аудит действий;
- **`retrain_history.jsonl`** — журнал попыток online-дообучения;
- **`labels_dataset.csv`** — накопленная разметка аналитиков для sandbox;
- **`sandbox_reports.jsonl`** — отчёты сравнения моделей;
- **`proxy_ingest_state.json`** — смещение при инкрементальном чтении NDJSON;
- **`train_reports/`** — отчёты об обучении, сравнения режимов detect и т.д.

### `src/`

**Исходный код библиотеки**: загрузка данных, признаки, модели, пайплайн детекции, корреляция с SIEM, online-буфер, governance. Импорты вида `from src....`.

### `scripts/`

**Исполняемые сценарии** с номерами (`01_prepare_data.py`, `02_train_all.py`, …). `main.py` по сути вызывает их с нужными аргументами. Прямой запуск скрипта даёт **больше опций**, чем обёртка в `main.py`.

### `dashboard/`

**Streamlit-приложение**: просмотр алертов, инцидентов, графики. Запуск через `python main.py dashboard`.

### `tests/`

Автотесты (`pytest`). Проверяют ключевую логику без ручного кликанья.

### `docs/`

Подробная документация: ТЗ кейса 4, трассировка требований, производительность, online на Windows и т.д. Если вы спорите с формулировкой «как в ТЗ», первичный источник — **`docs/TZ_CASE4.md`**, а не этот README.

### Прочее в корне (в т.ч. опциональная упаковка в exe)

- **`ids-cli.spec`**, **`ids-dashboard.spec`** — профили **PyInstaller**. Нужны только тем, кто хочет раздать преподавателю/стенду **готовые каталоги** `dist\ids-cli\`, `dist\ids-dashboard\` без установки Python на целевой машине. Сборка: из корня репозитория, с установленным `pyinstaller` и зависимостями (`pip install pyinstaller`), команды вида `pyinstaller --noconfirm --clean ids-cli.spec` и аналогично для `ids-dashboard.spec`. Отдельной папки `packaging/` в репозитории **нет** — сценарии сборки сведены к spec-файлам и этому README.
- **`dashboard_launcher.py`** — точка входа Streamlit при сборке дашборда через spec.
- **`Friday-WorkingHours.pcap`** (если есть локально) — пример трафика; большие датасеты часто **не** входят в git.

**Frozen CLI (`sys.frozen`):** если когда-нибудь запускаете собранный `ids-cli.exe`, а каталог с exe **нельзя записывать**, данные и копии конфигов по умолчанию переносятся в `%LOCALAPPDATA%\IDS_ML_Project` (см. `main.py`, переменная окружения **`IDS_PROJECT_ROOT`**). Дашборд в frozen-режиме ожидается **рядом**: `..\ids-dashboard\ids-dashboard.exe` относительно каталога с CLI.

## 4. Установка и с чего начать

Ниже — порядок **клонирование → версия Python → виртуальное окружение → установка пакетов → команды**. Все примеры с **`python`** и **`pip`** предполагают, что вы уже **активировали `.venv`** (или вызываете полный путь к `python.exe` из `.venv`).

### 4.1 Версия Python

В **`pyproject.toml`** задано **`requires-python = ">=3.11,<3.14"`** (совпадает с ожиданиями CI и наличием колёс PyTorch на Windows).

| Версия | Комментарий |
|--------|-------------|
| **3.12** (рекомендуется) | Удобно ставить torch и полный L2-стек под Windows. |
| **3.11** | Официально используется в GitHub Actions; полностью поддерживается. |
| **3.14** | Вне диапазона `pyproject`; глобальный 3.14 часто **без** пакетов проекта — не подменяйте им `.venv`. |

**Важно:** команды вроде `python main.py check` должны выполняться **тем же интерпретатором**, куда вы ставили `pip install -r requirements.txt`. Обычно это **`.\.venv\Scripts\python.exe`**, а не «первый попавшийся» Python из PATH.

### 4.2 Клонирование репозитория

```powershell
git clone https://github.com/Andrey20091/ids.git
cd ids
```

Если вы клонировали в другую папку (`ids-ml-project` и т.д.), везде ниже используйте **ваш** путь вместо `ids`.

### 4.3 Создание виртуального окружения `.venv`

Окружение изолирует пакеты проекта от системного Python и совпадает с настройкой **`.vscode/settings.json`** (интерпретатор по умолчанию в Cursor/VS Code).

**Автоматически** (если установлены **Python 3.12 или 3.11** и команда `py` их видит, например `py -0p`):

```powershell
cd путь\к\клону
powershell -ExecutionPolicy Bypass -File scripts\setup_venv.ps1
```

Скрипт создаст **`.venv`**, обновит `pip` и выполнит `pip install -r requirements.txt`.

**Вручную** (пример для 3.12):

```powershell
cd путь\к\клону
py -3.12 -m venv .venv
.\.venv\Scripts\python.exe -m pip install -U pip
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

Если папки **`.venv` ещё нет**, `main.py check` напомнит создать её (`py -3.12 -m venv .venv` и далее команды из подсказки).

### 4.4 Вход в окружение и работа «оттуда»

После создания `.venv` **активируйте** его в терминале — тогда команда **`python`** будет указывать на нужный интерпретатор.

**PowerShell** (из корня клона):

```powershell
.\.venv\Scripts\Activate.ps1
```

Если политика выполнения блокирует скрипт: выполните один раз для текущего пользователя  
`Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser`  
или запускайте без активации полным путём (см. ниже).

**cmd.exe:**

```bat
cd /d путь\к\клону
.venv\Scripts\activate.bat
```

**Без активации** (всегда работает, удобно в CI и при блокировке `Activate.ps1`):

```powershell
.\.venv\Scripts\python.exe main.py check
```

В корне есть **`run.cmd`**: он вызывает **`.\.venv\Scripts\python.exe`**, если `.venv` существует — можно писать `run.cmd main.py check`.

Дальше в этом README под **`python`** имеется в виду **интерпретатор из активированного `.venv`** (или явный путь к нему).

### 4.5 Опционально: PyTorch (AE, LSTM, embedding и т.д.)

После активации `.venv`:

```powershell
python -m pip install -r requirements-ml.txt
```

Для CPU-версии с индексом PyTorch (если обычный `pip` не находит колесо):

```powershell
python -m pip install -r requirements-ml.txt --index-url https://download.pytorch.org/whl/cpu
```

Если torch не нужен — достаточно RF/IF: в обучении используйте **`--skip-torch`** или **`--allow-partial`**, где поддерживается.

### 4.6 Кодировка консоли Windows

Если «кракозябры» вместо русского текста: **`chcp 65001`**, переменная **`PYTHONUTF8=1`**, или запуск через **`run_utf8.cmd`**.

### 4.7 Первые команды

```powershell
python main.py check          # проверка окружения
python main.py validate       # данные и модели на месте?
python main.py all --demo-mode   # демо: синтетика → prepare → detect (без обучения в demo)
```

Автоустановка зависимостей и базы GeoIP для подсказок по странам:

```powershell
python main.py bootstrap
```

### 4.8 Что важно знать после `git clone`

В репозитории **намеренно нет** большого **`data/processed/flows.csv`** (он в `.gitignore`). Зато есть **код**, **конфиги**, **готовые веса** в **`artifacts/`** и демо-сырьё в **`data/raw/`**.

**Минимальный рабочий сценарий для нового клона:**

1. §4.2–4.4: клон, **`.venv`**, **`pip install -r requirements.txt`** (и при необходимости §4.5 для torch).
2. **`python main.py all --demo-mode`** — синтетика, `prepare`, `detect`.
3. Либо свой CSV → **`python main.py prepare --input путь\к\файлу.csv`**.

**Почему `validate` может ругаться:** пока нет `flows.csv` по пути из `settings.yaml` — сначала шаг 2 или 3.

**Прокси после клона:** достаточно зависимостей и свободного порта; команды **`proxy`**, **`proxy-ingest`**, **`proxy-sync-buffer`** — в §6.

## 5. Конфигурация — зачем `settings.yaml` и что трогать в первую очередь

Файл **`config/settings.yaml`** — главный «пульт» без правки Python:

- **`paths`** — где лежат `flows_current`, `flows_online_buffer`, `artifacts`, файлы SIEM, водяные знаки online и т.д. Если вы перенесли проект на другой диск, правьте в первую очередь **paths**, а не код.
- **`threat_scoring.alert_threshold`** — число, от которого зависит, попадёт ли поток в список алертов после скоринга. Слишком высокий порог → «тишина» на демо; слишком низкий → много шума. Для стенда иногда используют **`python main.py detect --detect-demo-preset`** (см. §6).
- **`training_policy`** — кто имеет право на **baseline-train** (обычно только датасеты с тегом CICIDS), нужен ли **baseline manifest** перед online, запрещён ли полный `train` на «новых» данных без явной политики. Это защита от случайного смешения эталонного обучения и произвольного дампа.
- **`pipeline`**, **`online`**, **`aggregation`**, **`siem`** — поведение каскада L1/L2, интервал online, корреляция с SIEM.

Комментарии внутри YAML и **`docs/TZ_CASE4.md`** помогут, если нужно понять каждое поле. Не копируйте чужой `settings.yaml` вслепую: пути должны соответствовать **вашей** файловой системе.

## 6. Все команды `python main.py`

Общее правило: у **`main.py` один парсер аргументов**, поэтому **любой флаг можно набрать для любой команды**, но **неиспользуемые флаги просто игнорируются**. Ниже для каждой команды перечислены флаги, которые она **реально использует**.

Синтаксис:

```text
python main.py <команда> [флаги]
```

---

### `all` — демо-прогон «под ключ»

**Что делает:** проверка окружения → генерация синтетического CSV → **prepare** → **detect**. Обучение в этом режиме **намеренно не вызывается** (чтобы не путать demo и production lifecycle).

**Когда использовать:** быстро показать, что пайплайн живой.

**Обязательно:** **`--demo-mode`**. Без него команда завершится с подсказкой использовать `baseline-train` и отдельные шаги.

| Флаг | Зачем |
|------|--------|
| `--demo-mode` | Разрешает этот упрощённый сценарий. |
| `--dashboard` | После прогона запустить дашборд. |
| `--gen-seed` | Число для воспроизводимой «случайной» генерации демо-данных (по умолчанию 42). |
| `--gen-random-seed` | Каждый раз новый seed — данные каждый запуск другие. |
| `--input` | После генерации prepare читает **этот** файл (по умолчанию демо-путь из сценария генерации). |
| `--prepare-output` | Куда записать итоговый `flows.csv` (если не указано — как в `settings` / дефолт скрипта prepare). |
| `--prepare-append-output` | Дописать строки в конец выходного файла вместо полной перезаписи. |
| `--prepare-features-yaml` | Альтернативный список признаков. |
| `--header-bytes-npz` | Доп. данные байтов заголовков для CNN-ветки (если используете). |
| `--prepare-no-cicids-normalize` | Отключить нормализацию в стиле CICIDS. |
| `--prepare-pcap-enrichment` | Путь к PCAP для обогащения DNS/HTTP признаками. |

---

### `check` — проверка окружения

**Что делает:** скрипт проверяет зависимости, версии, доступность ключевых путей (детали — вывод в консоль).

**Флаги:** только глобальные, специальных нет. Перед запуском печатается текущий **`alert_threshold`** из настроек.

---

### `bootstrap` — подготовка окружения «как для продакшн-лабы»

**Что делает:** ставит недостающие пакеты по необходимости, подтягивает **MMDB** для геолокации IP (DB-IP Lite и т.п. — см. скрипт).

**Флаги:** специальных у `main.py` нет.

---

### `validate` — здоровье проекта

**Что делает:** проверяет согласованность путей, наличие важных файлов, разметки и артефактов (`scripts/validate_project_state.py`).

**Флаги:** нет.

---

### `generate` — только синтетические сырые данные

**Что делает:** создаёт CSV с имитацией CICIDS-подобных потоков для тестов.

| Флаг | Зачем |
|------|--------|
| `--gen-seed` | Фиксированный seed. |
| `--gen-random-seed` | Случайный seed каждый раз. |

### `prepare` — превращает сырой CSV в таблицу признаков для ML

**Что делает:** нормализует имена колонок, заполняет пропуски, считает признаки → типичный выход **`data/processed/flows.csv`**.

| Флаг | Зачем |
|------|--------|
| `--input` | Входной сырой CSV (по умолчанию демо-файл). |
| `--prepare-output` | Куда записать результат (по умолчанию — из настроек / дефолт скрипта). |
| `--prepare-append-output` | Режим **дописывания** в конец файла с проверкой схемы (без второго заголовка). |
| `--prepare-features-yaml` | Свой YAML признаков. |
| `--header-bytes-npz` | NPZ с байтами заголовков для соответствующих моделей. |
| `--prepare-no-cicids-normalize` | Без нормализации CICIDS. |
| `--prepare-pcap-enrichment` | PCAP для дополнительных признаков по plaintext DNS/HTTP. |

### `ingest-new-data` — то же, что prepare, но семантика «новая порция трафика»

### `proxy` — локальный HTTP(S) прокси для записи трафика

**Что делает:** поднимает прокси; браузер или система направляет трафик → события пишутся в **NDJSON**.

| Флаг | Зачем |
|------|--------|
| `--proxy-bind` | IP интерфейса (часто `127.0.0.1`). |
| `--proxy-port` | Порт (по умолчанию 8899). |
| `--proxy-output` | Файл NDJSON лога. |
| `--proxy-host-filter` | Записывать только хосты, содержащие подстроку. |

## 8. Тесты, опциональная сборка exe, сценарии и типичные проблемы

### Тесты

Автотесты проверяют политику обучения, детекцию, прокси, SIEM, online-буфер и многое другое. Запуск из корня:

```powershell
python -m pytest tests -q --tb=short
```

Если тест падает у вас, но проходит в CI, чаще всего причина — **другой интерпретатор** (не тот venv) или **устаревшие артефакты** в `artifacts/` после экспериментов. Сравните `python --version` и путь к `python` с тем, что использует IDE.

### Опционально: сборка portable exe (PyInstaller)

Репозиторий **не содержит** готовых `dist\` и **не обязан** собираться в exe для учебной работы — достаточно `python main.py`. Exe имеет смысл, если нужно отдать **одну папку** без установки Python на чужой ПК.

1. В активированном venv: `pip install pyinstaller` (и все зависимости проекта, включая torch, если нужен полный CLI внутри bundle).
2. Из **корня** репозитория:

```powershell
pyinstaller --noconfirm --clean ids-cli.spec
pyinstaller --noconfirm --clean ids-dashboard.spec
```

3. Проверка: `.\dist\ids-cli\ids-cli.exe check` и `.\dist\ids-cli\ids-cli.exe -h`.

Spec-файлы уже настроены под структуру проекта; при добавлении новых модулей в точках входа иногда нужно обновить `datas`/`hiddenimports` в spec — смотрите сообщения PyInstaller при сборке.

### Готовые сценарии Windows (`.cmd` / `.ps1`)

- **`run_corporate_e2e_example.cmd`** — валидация корпоративного CSV, prepare, train с `--skip-torch`, короткий detect.
- **`run_pre_demo_smoke.cmd`** — быстрый smoke перед демонстрацией.
- **`run_utf8.cmd`** — обёртка с UTF-8 для консоли, если русский текст ломается.

Запускайте `.cmd` двойным щелчком или из `cmd.exe`; в PowerShell иногда мешает политика выполнения — тогда `cmd /c run_....cmd`.

### Типичные проблемы и что проверить

| Симптом | Что проверить |
|--------|----------------|
| «Нет модели RF» / `validate` ругается на артефакты | Есть ли `artifacts/rf_model.joblib` после клона; не удалили ли вы `artifacts/` локально. |
| `detect` не находит CSV | Путь в `settings.paths.flows_current` или флаг `--detect-data`; после клона сначала `prepare` или `all --demo-mode`. |
| «Кракозябры» в консоли Windows | `chcp 65001`, `PYTHONUTF8=1`, или `run_utf8.cmd`. |
| Torch / AE / LSTM не работают | Установлен ли torch **в том же** venv, откуда вызываете `main.py`; иначе `--skip-torch` / `--allow-partial`. |
| Прокси не пишет лог | Права на каталог `data/raw`, не занят ли порт, не остался ли `.lock` от старого процесса. |
| SIEM NDJSON «ломает» детекцию | С версии с исправлением битые строки NDJSON **пропускаются**; убедитесь, что обновили репозиторий. |
