## 1. Что это за проект

**IDS ML Project** — это учебно-демонстрационная система **обнаружения сетевых угроз** с помощью машинного обучения. Она не «лезет в ваш интернет сама»: вы подаёте на вход **записи о сетевых потоках** (файлы CSV в стиле датасета CICIDS2017), либо **собираете трафик** через локальный прокси или из PCAP. Программа:

- приводит данные к единому виду (**подготовка признаков**);
- **обучает** несколько моделей (в т.ч. Random Forest, Isolation Forest; опционально — нейросети, если установлен PyTorch);
- **прогоняет детекцию** и выдаёт **алерты** (подозрительная активность);
- умеет **дообучаться по новым данным** (online), вести **инциденты**, сравнивать модели в **песочнице** и показывать всё в **веб-дашборде**.

Это **не замена коммерческому IPS** и не гарантия «ловли всего в реальном времени на гигабитах»: это полный **сквозной пайплайн** для лабораторных работ и стендов.

## 2. Как это работает одним абзацем

**Сырой CSV** (или NDJSON с прокси, или PCAP) → скрипт **prepare** строит таблицу признаков **`flows*.csv`** → **train / baseline-train** сохраняют модели в **`artifacts/`** → **detect** читает потоки и пишет **`storage/alerts_latest.json`** → дашборд и инциденты читают эти файлы. Параллельно **online** может подстраивать модели по накапливающемуся буферу **`flows_online_buffer.csv`**, не обязательно перезаписывая основной снимок **`flows.csv`**.

## 3. Структура папок — что где лежит

| Папка / файл | Назначение |
|----------------|------------|
| **`main.py`** | Единая точка входа: все подкоманды (`detect`, `prepare`, `proxy`, …). Запускайте из корня проекта. |
| **`pyproject.toml`** | Метаданные пакета и зависимости для установки через `pip install -e .`. |
| **`requirements.txt`** | Базовые зависимости (pandas, sklearn, PyYAML, …). |
| **`requirements-ml.txt`** | Опционально: PyTorch и всё для «глубоких» моделей. |
| **`.env.example`** | Пример переменных окружения (если используются в вашей среде). |
| **`run.cmd`**, **`run_utf8.cmd`**, **`run.ps1`** | Удобные обёртки запуска под Windows (в т.ч. UTF-8 для консоли). |
| **`run_*_example.cmd` / `.ps1`** | Готовые сценарии: корпоративный e2e, полный ТЗ-пятница и т.д. |

### `config/`

Настройки **без правки кода**:

- **`settings.yaml`** — пути к файлам данных (`flows.csv`, буфер онлайн, SIEM, пороги угроз, политика обучения, интервалы online и др.).
- **`feature_columns.yaml`** — какие колонки и производные признаки участвуют в ML.
- Другие YAML — вспомогательные схемы (например каноника числовых полей CICIDS).

### `data/`

- **`raw/`** — «как пришло»: синтетические демо-файлы, выгрузки CICIDS (часть нужно скачать отдельно), NDJSON прокси, PCAP-конверсии. Внутри есть маленькие **`README.md`** по подпапкам.
- **`processed/`** — результат **prepare**: основной **`flows.csv`**, большие **`flows_baseline.csv`**, **`flows_online_buffer.csv`** для потокового контура и т.п.

### `artifacts/`

**Обученные модели** и реестр:

- файлы вида **`rf_model.joblib`**, **`if_model*.joblib`**, нейросетевые веса — в зависимости от того, что вы обучили;
- **`model_registry.json`** — какая версия моделей «активна», какие наборы есть для sandbox / deploy.

Не коммитьте сюда гигантские бинарники, если политика git это запрещает; локально это рабочие файлы.

### `storage/`

**Рабочее состояние приложения** (алерты, истории, чекпоинты):

- **`alerts_latest.json`** — последние алерты для дашборда;
- **`incidents.jsonl`**, **`incident_actions.jsonl`** — инциденты и аудит действий;
- **`retrain_history.jsonl`** — журнал попыток online-дообучения;
- **`labels_dataset.csv`** — накопленная разметка аналитиков для sandbox;
- **`sandbox_reports.jsonl`** — отчёты сравнения моделей;
- **`proxy_ingest_state.json`** — смещение при инкрементальном чтении NDJSON;
- **`train_reports/`** — отчёты об обучении, сравнения режимов detect и т.д.

### `src/`

**Исходный код библиотеки**: загрузка данных, признаки, модели, пайплайн детекции, корреляция с SIEM, online-буфер, governance. Импорты вида `from src....`.

### `scripts/`

**Исполняемые сценарии** с номерами (`01_prepare_data.py`, `02_train_all.py`, …). `main.py` по сути вызывает их с нужными аргументами. Прямой запуск скрипта даёт **больше опций**, чем обёртка в `main.py`.

### `dashboard/`

**Streamlit-приложение**: просмотр алертов, инцидентов, графики. Запуск через `python main.py dashboard`.

### `tests/`

Автотесты (`pytest`). Проверяют ключевую логику без ручного кликанья.

### `docs/`

Подробная документация: ТЗ кейса 4, трассировка требований, производительность, online на Windows и т.д.

### `packaging/`

Сборка **исполняемых файлов** (`ids-cli.exe`, `ids-dashboard.exe`) через PyInstaller. Подробности — в **`packaging/README.md`**.

### Прочее в корне

- **`ids-cli.spec`**, **`ids-dashboard.spec`** — профили PyInstaller.
- **`dashboard_launcher.py`** — точка входа для сборки дашборда.
- **`Friday-WorkingHours.pcap`** (если есть) — пример трафика; большие датасеты часто **не** входят в git.

## 4. Установка и с чего начать

### 4.1 Виртуальное окружение и пакеты

```powershell
cd путь\к\ids-ml-project
python -m venv .venv
.\.venv\Scripts\activate
python -m pip install -r requirements.txt
```

Опционально PyTorch (CPU), если нужны автоэнкодер / LSTM и полный стек:

```powershell
python -m pip install -r requirements-ml.txt --index-url https://download.pytorch.org/whl/cpu
```

Если torch не ставится — можно пользоваться **только Random Forest + Isolation Forest**, передавая **`--skip-torch`** или **`--allow-partial`** там, где это поддержано.

### 4.2 Кодировка консоли Windows

Если «кракозябры» вместо русского текста: включите UTF-8 (`chcp 65001`, `PYTHONUTF8=1`) или запускайте через **`run_utf8.cmd`**.

### 4.3 Первые команды

```powershell
python main.py check          # проверка окружения
python main.py validate       # данные и модели на месте?
python main.py all --demo-mode   # демо: синтетика → prepare → detect (без обучения в demo)
```

Автоустановка зависимостей и базы GeoIP для подсказок по странам:

```powershell
python main.py bootstrap
```

## 5. Конфигурация в двух словах

- Все **пути к файлам** по возможности задавайте в **`config/settings.yaml`** (`paths`), а не хардкодьте в коде.
- **Порог алерта** (`threat_scoring.alert_threshold`) влияет на то, что попадёт в алерты при детекции.
- **Политика обучения** (`training_policy`) ограничивает, на каких данных можно первый «базовый» train и когда разрешён полный retrain — чтобы не смешивать эталонный CICIDS и произвольный трафик без явного намерения.

Детали полей — в комментариях внутри YAML и в **`docs/TZ_CASE4.md`**.

## 6. Все команды `python main.py`

Общее правило: у **`main.py` один парсер аргументов**, поэтому **любой флаг можно набрать для любой команды**, но **неиспользуемые флаги просто игнорируются**. Ниже для каждой команды перечислены флаги, которые она **реально использует**.

Синтаксис:

```text
python main.py <команда> [флаги]
```

---

### `all` — демо-прогон «под ключ»

**Что делает:** проверка окружения → генерация синтетического CSV → **prepare** → **detect**. Обучение в этом режиме **намеренно не вызывается** (чтобы не путать demo и production lifecycle).

**Когда использовать:** быстро показать, что пайплайн живой.

**Обязательно:** **`--demo-mode`**. Без него команда завершится с подсказкой использовать `baseline-train` и отдельные шаги.

| Флаг | Зачем |
|------|--------|
| `--demo-mode` | Разрешает этот упрощённый сценарий. |
| `--dashboard` | После прогона запустить дашборд. |
| `--gen-seed` | Число для воспроизводимой «случайной» генерации демо-данных (по умолчанию 42). |
| `--gen-random-seed` | Каждый раз новый seed — данные каждый запуск другие. |
| `--input` | После генерации prepare читает **этот** файл (по умолчанию демо-путь из сценария генерации). |
| `--prepare-output` | Куда записать итоговый `flows.csv` (если не указано — как в `settings` / дефолт скрипта prepare). |
| `--prepare-append-output` | Дописать строки в конец выходного файла вместо полной перезаписи. |
| `--prepare-features-yaml` | Альтернативный список признаков. |
| `--header-bytes-npz` | Доп. данные байтов заголовков для CNN-ветки (если используете). |
| `--prepare-no-cicids-normalize` | Отключить нормализацию в стиле CICIDS. |
| `--prepare-pcap-enrichment` | Путь к PCAP для обогащения DNS/HTTP признаками. |

### `check` — проверка окружения

**Что делает:** скрипт проверяет зависимости, версии, доступность ключевых путей (детали — вывод в консоль).

**Флаги:** только глобальные, специальных нет. Перед запуском печатается текущий **`alert_threshold`** из настроек.

### `bootstrap` — подготовка окружения «как для продакшн-лабы»

**Что делает:** ставит недостающие пакеты по необходимости, подтягивает **MMDB** для геолокации IP (DB-IP Lite и т.п. — см. скрипт).

**Флаги:** специальных у `main.py` нет.

### `validate` — здоровье проекта

**Что делает:** проверяет согласованность путей, наличие важных файлов, разметки и артефактов (`scripts/validate_project_state.py`).

**Флаги:** нет.

### `generate` — только синтетические сырые данные

**Что делает:** создаёт CSV с имитацией CICIDS-подобных потоков для тестов.

| Флаг | Зачем |
|------|--------|
| `--gen-seed` | Фиксированный seed. |
| `--gen-random-seed` | Случайный seed каждый раз. |

### `prepare` — превращает сырой CSV в таблицу признаков для ML

**Что делает:** нормализует имена колонок, заполняет пропуски, считает признаки → типичный выход **`data/processed/flows.csv`**.

| Флаг | Зачем |
|------|--------|
| `--input` | Входной сырой CSV (по умолчанию демо-файл). |
| `--prepare-output` | Куда записать результат (по умолчанию — из настроек / дефолт скрипта). |
| `--prepare-append-output` | Режим **дописывания** в конец файла с проверкой схемы (без второго заголовка). |
| `--prepare-features-yaml` | Свой YAML признаков. |
| `--header-bytes-npz` | NPZ с байтами заголовков для соответствующих моделей. |
| `--prepare-no-cicids-normalize` | Без нормализации CICIDS. |
| `--prepare-pcap-enrichment` | PCAP для дополнительных признаков по plaintext DNS/HTTP. |

---

### `ingest-new-data` — то же, что prepare, но семантика «новая порция трафика»

**Что делает:** вызывает тот же prepare-пайплайн; удобно для сценария «приехал новый дамп → дописали / перезаписали flows».

**Флаги:** те же, что у **`prepare`**, включая **`--input`**, **`--prepare-output`**, **`--prepare-append-output`** / **`--ingest-append-output`** (полное имя и короткий alias одного и того же поведения).

---

### `baseline-train` — первичное обучение на эталонном датасете (например CICIDS2017)

**Что делает:** обучает модели с пометкой «baseline» согласно **`training_policy`** в `settings.yaml` (нужны корректные `--dataset-tag` и путь к подготовленным потокам).

| Флаг | Зачем |
|------|--------|
| `--baseline-data` | Путь к prepared CSV для baseline (приоритетный источник данных для этого режима). |
| `--train-data` | Альтернатива: тоже путь к flows, если так удобнее передать. |
| `--dataset-tag` | Метка датасета для политики (часто `cicids2017`). |
| `--dataset-source` | Строка-описание источника для отчётов. |
| `--baseline-features-yaml` | Отдельный YAML признаков для этого прогона. |
| `--training-profile` | `production` или `development` — профили эпох/размеров из `settings.yaml`. |
| `--skip-torch` | Не обучать тяжёлые нейросети даже если torch установлен. |
| `--force-rebaseline` | Разрешить перезапись baseline только если это позволено настройками политики. |

---

### `train` — полное переобучение (часто ограничено политикой)

**Что делает:** запускает полный цикл обучения как в скрипте **`02_train_all.py`**; на «чужих» данных без baseline может быть **заблокировано** политикой — это нормально.

| Флаг | Зачем |
|------|--------|
| `--train-data` | Путь к prepared flows. |
| `--dataset-tag` | Для gate политики. |
| `--dataset-source` | Описание источника. |
| `--training-profile` | production / development. |
| `--skip-torch` | Без AE/LSTM и т.д. |
| `--allow-partial` | Если torch недоступен, не падать на старте — только RF/IF (для `train`/`all` логика см. сообщения CLI). |
| `--force-rebaseline` | Условный перезапуск baseline при разрешении в конфиге. |

Для команд **`all`** и **`train`**: если torch не импортируется, нужен **`--skip-torch`** или **`--allow-partial`**, иначе программа выйдет с ошибкой.

### `detect` — пакетная детекция по файлу потоков

**Что делает:** читает подготовленный CSV потоков, прогоняет каскад моделей и записывает алерты (по умолчанию **`storage/alerts_latest.json`**).

| Флаг | Зачем |
|------|--------|
| `--detect-data` | Явный путь к flows CSV (иначе дефолт скрипта детекции). |
| `--detect-use-online-buffer` | Взять путь из **`paths.flows_online_buffer`** в `settings.yaml` (прокси-буфер). |
| `--detect-new-rows-only` | Обработать только строки после последнего запуска (инкремент). |
| `--detect-incremental-state` | Файл состояния для инкремента (по умолчанию под `storage/`). |
| `--detect-limit` | Максимум строк (ускорение тестов). |
| `--detect-parquet-cache` | Кэш в parquet рядом с CSV для ускорения повторных прогонов. |
| `--detect-csv-engine` | `c` или `pyarrow` — движок чтения CSV. |
| `--detect-benchmark` | Профилирование после прогона. |
| `--detect-parallel-l2` | Тяжёлый L2 для всех потоков (режим отчёта; см. ТЗ). |
| `--detect-packet-lstm-scores` | NPZ с оценками packet-LSTM для расширенного режима. |
| `--detect-stream-chunk-rows` | Читать CSV чанками (экономия памяти). |
| `--detect-log-wall-time` | Писать в лог реальное время прохода. |
| `--detect-compare-modes-report` | JSON-отчёт сравнения режимов детекции. |
| `--detect-compare-report-path` | Куда сохранить этот отчёт. |
| `--detect-features-yaml` | Свой набор признаков. |
| `--detect-demo-preset` | Мягкий режим для стенда: больше «видимости» алертов. |
| `--detect-dedup-window-seconds` | Не дублировать алерты по IP/severity в окне N секунд. |
| `--detect-disable-proxy-rules` | Отключить слияние правил и ML для proxy-подобных признаков. |

### `clear-online-buffer` — обнулить онлайн-буфер

**Что делает:** обрезает файл буфера до заголовка, сбрасывает watermark и связанное состояние (см. **`scripts/clear_online_buffer.py`**).

| Флаг | Зачем |
|------|--------|
| `--clear-buffer` | Явный путь к CSV буфера; если пусто — берётся **`paths.flows_online_buffer`** из настроек. |

### `online` — одно или многократное дообучение по новым данным

**Что делает:** запускает цикл online-retrain (по умолчанию читает **`flows_online_buffer`** из настроек).

| Флаг | Зачем |
|------|--------|
| `--online-data` | Другой входной CSV вместо буфера по умолчанию. |
| `--online-loop` | Крутиться бесконечно по таймеру из настроек. |
| `--online-delayed-first-tick` | В loop не выполнять первый тик сразу, а ждать интервал. |

### `realtime` — «почти онлайн» детекция по мере поступления данных

**Что делает:** периодически читает буфер/файл потоков и обновляет алерты; предназначено для демонстрации near-realtime.

| Флаг | Зачем |
|------|--------|
| `--realtime-data` | Источник потоков (иначе буфер из настроек). |
| `--realtime-output-alerts` | Куда писать JSON алертов. |
| `--realtime-max-alerts` | Сколько последних алертов хранить в файле. |
| `--realtime-iterations` | Число циклов (0 = бесконечно). |
| `--realtime-features-yaml` | Свой YAML признаков. |
| `--realtime-poll-seconds` | Как часто просыпаться. |
| `--realtime-batch-size` | Размер батча за итерацию. |
| `--realtime-auto-online-retrain` | В том же процессе вызывать online-retrain. |
| `--realtime-auto-online-every-iters` | Retrain каждые N итераций цикла. |

### `proxy` — локальный HTTP(S) прокси для записи трафика

**Что делает:** поднимает прокси; браузер или система направляет трафик → события пишутся в **NDJSON**.

| Флаг | Зачем |
|------|--------|
| `--proxy-bind` | IP интерфейса (часто `127.0.0.1`). |
| `--proxy-port` | Порт (по умолчанию 8899). |
| `--proxy-output` | Файл NDJSON лога. |
| `--proxy-host-filter` | Записывать только хосты, содержащие подстроку. |
| `--proxy-max-log-mb` | Ротация лога при размере в МБ (0 = выкл). |
| `--proxy-max-log-backups` | Сколько старых файлов хранить при ротации. |

Рядом с логом может создаваться **lock-файл** — не запускайте два прокси в один файл.

---

### `proxy-ingest` — NDJSON → CSV (+ опционально prepare)

**Что делает:** преобразует записи прокси в таблицу, похожую на CICIDS, для дальнейшего **`prepare`**.

| Флаг | Зачем |
|------|--------|
| `--ingest-ndjson` | Входной NDJSON. |
| `--ingest-csv-out` | Выходной CSV. |
| `--ingest-state-file` | JSON с позицией чтения для инкремента. |
| `--ingest-incremental` | Читать только новые записи после checkpoint. |
| `--ingest-append` | Дописывать в CSV, а не перезаписывать. |
| `--ingest-prepare` | После конвертации сразу вызвать prepare с **`--input`** = этот CSV; дальше работают **`--prepare-output`**, **`--prepare-append-output`** и остальные флаги prepare. |

---

### `proxy-sync-buffer` — инкрементальный ingest + дописывание в онлайн-буфер

**Что делает:** сочетает инкрементальный **`proxy-ingest`** (по умолчанию state **`storage/proxy_ingest_state.json`**) и **prepare в append** в путь **`flows_online_buffer`** из настроек — удобный один шаг для сценария «прокси крутится → в буфере копятся признаки».

**Флаги:** как у **`proxy-ingest`** и **prepare** (`--ingest-*`, `--prepare-output` если переопределяете путь; по умолчанию выход prepare подставляется на online buffer автоматически в коде `main.py`).

### `pcap-flows` — из PCAP в таблицу потоков

**Что делает:** нужен **scapy**; собирает потоки в CSV, опционально сразу **`prepare`**.

| Флаг | Зачем |
|------|--------|
| `--pcap` | Путь к `.pcap` / `.pcapng` (обязателен). |
| `--pcap-output` | Куда сохранить сырой CSV потоков. |
| `--pcap-prepare` | После конвертации запустить prepare (используются флаги **`--input`** через этот CSV и prepare-флаги). |

## 7. Справочник флагов по темам

Здесь те же флаги, но сгруппированы по смыслу — удобно искать «что вообще есть».

### Данные и prepare

- **`--input`** — сырой CSV для prepare / цепочек, где prepare читает файл после конвертации.
- **`--prepare-output`** — итоговый flows CSV.
- **`--prepare-append-output`**, **`--ingest-append-output`** — дописать строки в конец.
- **`--prepare-features-yaml`** — альтернативный список признаков.
- **`--header-bytes-npz`** — байтовые признаки для части моделей.
- **`--prepare-no-cicids-normalize`** — отключить нормализацию CICIDS.
- **`--prepare-pcap-enrichment`** — PCAP для обогащения.

### Обучение

- **`--train-data`**, **`--baseline-data`** — пути к подготовленным данным.
- **`--dataset-tag`**, **`--dataset-source`** — метки для политики и отчётов.
- **`--baseline-features-yaml`** — YAML только для baseline.
- **`--training-profile`** — `production` / `development`.
- **`--skip-torch`** — без глубоких моделей.
- **`--allow-partial`** — работать без torch там, где это разрешено.
- **`--force-rebaseline`** — попытка перезаписи baseline при соблюдении правил в конфиге.

### Детекция

См. команду **`detect`**: `--detect-*` (данные, лимиты, кэш, режимы L2, demo, дедуп, отключение proxy-rules).

### Online и буфер

- **`--online-data`**, **`--online-loop`**, **`--online-delayed-first-tick`**.
- **`--clear-buffer`** — для **`clear-online-buffer`**.

### Прокси и ingest

- **`--proxy-*`**, **`--ingest-*`** — см. команды **`proxy`**, **`proxy-ingest`**, **`proxy-sync-buffer`**.

### PCAP

- **`--pcap`**, **`--pcap-output`**, **`--pcap-prepare`**.

### Инциденты и разметка

- **`--incident-*`**, **`--labels-input`**, **`--labels-output`**, **`--report-limit`**.

### Governance

- **`--candidate-model-set-id`**, **`--sandbox-min-delta-f1`**, **`--sandbox-min-precision`**, **`--model-set-id`**, **`--approved-by`**.
